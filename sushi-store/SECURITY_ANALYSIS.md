# 🔒 Анализ безопасности Sushi Store

## ⚠️ **КРИТИЧЕСКИЕ УЯЗВИМОСТИ БЕЗОПАСНОСТИ**

### 🚨 **1. ХРАНЕНИЕ ПАРОЛЕЙ В ОТКРЫТОМ ВИДЕ**
**Уровень угрозы: КРИТИЧЕСКИЙ**

```json
// server/db.json
"users": [
  {
    "id": "admin1",
    "phone": "+7 (999) 123-45-67",
    "password": "admin123",  // ❌ ПАРОЛЬ В ОТКРЫТОМ ВИДЕ!
    "role": "admin"
  }
]
```

**Проблема:** Пароли хранятся в открытом виде в JSON файле
**Последствия:** 
- Любой, кто получит доступ к файлу, увидит пароли
- При утечке данных все аккаунты скомпрометированы
- Невозможно восстановить пароли

**Решение:** Использовать хеширование паролей (bcrypt, scrypt)

---

### 🚨 **2. СЛАБАЯ ГЕНЕРАЦИЯ ТОКЕНОВ**
**Уровень угрозы: ВЫСОКИЙ**

```javascript
// server/server.js:291
const token = crypto.randomBytes(16).toString('hex');
```

**Проблема:** Токены генерируются только 16 байтами (32 символа hex)
**Последствия:**
- Возможность брутфорса токенов
- Короткий срок жизни увеличивает риск

**Решение:** Увеличить до 32+ байт, добавить время жизни токенов

---

### 🚨 **3. ОТСУТСТВИЕ HTTPS**
**Уровень угрозы: ВЫСОКИЙ**

**Проблема:** Все данные передаются по HTTP
**Последствия:**
- Перехват паролей и токенов
- Man-in-the-middle атаки
- Подмена данных

**Решение:** Обязательно использовать HTTPS в продакшене

---

### 🚨 **4. CORS НАСТРОЕН НА "*"**
**Уровень угрозы: СРЕДНИЙ**

```javascript
// server/server.js:254
'Access-Control-Allow-Origin': '*'
```

**Проблема:** Разрешен доступ с любых доменов
**Последствия:**
- CSRF атаки
- Утечка данных через сторонние сайты

**Решение:** Ограничить конкретными доменами

## 🔍 **ДЕТАЛЬНЫЙ АНАЛИЗ АТАК**

### ❌ **Может ли хакер подменить роль?**

**Клиентская сторона:** ❌ **ДА, легко**
```javascript
// В браузере можно изменить
localStorage.setItem('role', 'admin');
auth.role = 'admin';
```

**Серверная сторона:** ✅ **НЕТ, защищено**
```javascript
// Сервер проверяет токен и получает роль из сессии
const user = getUserFromAuth(req);
if (!user || user.role !== 'admin') {
  return sendJSON(res, 403, { error: 'Forbidden' });
}
```

### ❌ **Может ли хакер увидеть заказы?**

**Прямой доступ к API:** ❌ **НЕТ, защищено**
```javascript
// /api/orders требует админскую роль
if (!user || user.role !== 'admin') {
  return sendJSON(res, 403, { error: 'Forbidden' });
}
```

**Но может:** ✅ **ДА, если получит валидный админский токен**

### ❌ **Может ли хакер редактировать заказы?**

**Прямой доступ к API:** ❌ **НЕТ, защищено**
```javascript
// Все операции с заказами требуют админскую роль
if (!user || user.role !== 'admin') {
  return sendJSON(res, 403, { error: 'Forbidden' });
}
```
